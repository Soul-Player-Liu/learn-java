# 文档体系标准

本文是给大型业务项目复用的文档体系标准。它借鉴 harness engineering 的思路：仓库文档不是静态百科，而是能帮助人和智能体找到上下文、执行验证、完成回归的工程记录系统。

参考来源：[OpenAI - Harness engineering: leveraging Codex in an agent-first world](https://openai.com/index/harness-engineering/)。

## 核心原则

- 地图优先：入口文档只做导航，不把所有细节塞进一个文件。
- 仓库为准：重要工程知识应沉淀在仓库或连接系统中，而不是散落在聊天记录里。
- 分层加载：新人或智能体先读短入口，再按任务进入更深文档。
- 可验证：规范要尽量指向代码、脚本、测试、CI job 或生成产物。
- 少重复：同一命令和规则不要在多个地方长篇复制。
- 可迁移：标准文档写方法论和检查点，不把示例项目的临时选择写成硬规定。

## 推荐结构

```text
.
├── README.md
├── AGENTS.md
└── docs/
    ├── INDEX.md
    ├── ARCHITECTURE.md
    ├── standards/
    │   ├── monorepo.md
    │   ├── testing.md
    │   ├── mock.md
    │   └── documentation.md
    ├── decisions/
    ├── runbooks/
    ├── business/
    ├── harness/
    └── schema/
```

必选和可选边界：

- `README.md` 通常必选，但必须短。
- `AGENTS.md` 适合有智能体协作或多人接手的大项目。
- `docs/INDEX.md` 适合文档超过三份后引入。
- `docs/ARCHITECTURE.md` 适合有多个模块或服务时引入。
- `docs/standards/` 用于规范模板，不要求业务项目机械复制所有文件。
- `docs/decisions/` 用于 ADR，只有高成本或不可逆决策才需要。
- `docs/runbooks/` 用于常见操作和故障处理。
- `docs/business/` 用于真实业务流程、状态机、术语和关键表。
- `docs/harness/` 用于 agent task、评测用例、回归清单和自动化基准。
- `docs/schema/` 用于可生成、可检查漂移的数据库证据。

## README 标准

README 只回答：

- 这个仓库是什么。
- 主要技术栈和模块是什么。
- 文档从哪里读。
- 如何启动。
- 如何跑最常用验证。

README 不应该承载：

- 完整测试体系解释。
- 完整 mock 规范。
- 长篇架构讨论。
- 历史决策过程。
- 大量排障细节。

## AGENTS 标准

`AGENTS.md` 是给智能体和新接手工程师的短导航，应控制在可快速读完的长度。它应该指向 deeper docs，而不是复制全部内容。

适合包含：

- 首读路线。
- 修改边界。
- 常用验证命令。
- 高风险改动提醒。

不适合包含：

- 完整业务文档。
- 所有接口说明。
- 所有测试细节。
- 长篇背景材料。

## Architecture 标准

`docs/ARCHITECTURE.md` 记录当前系统事实，不写成抽象教程。

建议包含：

- 模块图。
- 模块职责。
- 依赖方向。
- 契约流。
- 数据流。
- mock 流。
- 质量门禁。
- 当前重要取舍。

架构文档应随代码更新。不能验证或已经过期的架构图，比没有文档更危险。

## Standards 标准

`docs/standards/` 放可迁移规范，面向后续大项目复用。

写法要求：

- 先写原则，再写推荐结构。
- 区分“标准要求”和“本仓库示例”。
- 每条关键要求尽量有验证方式。
- 不把一个小项目的规模误写成所有项目必须维护的文档清单。
- 不用“必须维护这些文件”这类机械说法；改成“推荐结构”和“按规模选择”。

## Harness 文档

当项目开始使用智能体长期改代码时，应考虑建立 `docs/harness/`：

- `tasks/`：可重复执行的任务模板。
- `evals/`：评测用例和通过标准。
- `rubrics/`：代码审查和完成度评分规则。
- `reports/`：周期性回归或评测结果。

这些文档应与 CI、测试和脚本相连。只写提示词但没有可验证结果，不算完整 harness。

## 文档评审清单

提交文档前检查：

- README 是否仍然短。
- AGENTS 是否是地图而不是百科全书。
- 新规范是否放在 `docs/standards/`。
- 架构事实是否有代码或测试证据。
- 命令是否真实存在。
- 文件路径是否正确。
- 是否删除了过期重复内容。
- 是否明确区分本仓库示例和大项目标准。
- 是否能帮助后续智能体少读无关上下文。
